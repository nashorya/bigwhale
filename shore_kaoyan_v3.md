## 1. 架构设计

上岸 采用**四层解耦架构**，v3.0 在中央调度层新增 Persona Engine，负责将人物卡风格注入所有对外输出。

```
┌─────────────────────────────────────────────────────┐
│                     通信层                           │
│            OneBot 11 协议适配器（QQ I/O）            │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                  中央调度层                          │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │   Manager   │  │  Scheduler   │  │  Persona   │ │
│  │  跨科权重   │  │  日程引擎    │  │  Engine    │ │
│  │  进度感知   │  │  推词时机    │  │  角色渲染  │ │
│  └─────────────┘  └──────────────┘  └────────────┘ │
│  ┌──────────────────────────────────────────────┐   │
│  │           Emotion Detector（v3.0）           │   │
│  │      情绪信号检测 · 触发陪伴流程              │   │
│  └──────────────────────────────────────────────┘   │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                    存储层                            │
│   kaoyan_system.db（SQLite 3）                      │
│   subjects · knowledge_points · user_schedule       │
│   persona_config · emotion_log（v3.0 新增）         │
└─────────────────────────────────────────────────────┘
```

### 1.1 Persona Engine 工作原理

所有对外推送内容在发出前，统一经过 Persona Engine 的**风格渲染层**：

```
原始内容（模板文本）
  → Persona Engine 读取 persona_config.active_persona
  → 查询对应角色的 tone_profile（口吻参数）
  → 替换称呼、调整语气词、注入角色口癖
  → 输出最终消息
```

Persona Engine 对调度逻辑**完全透明**——Manager 和 Scheduler 只生产内容，不感知角色，角色切换不影响任何业务逻辑。

## 3. 核心功能

### A. 全方位进度感知（Multi-Subject Tracker）

#### 3A-1 跨学科掌握度调度

Manager 每隔 30 分钟计算一次各科加权均分：

```
科目均分 = Σ(importance_i × mastery_level_i) / Σ(importance_i)
```

当某科均分 < 2.5 且与最高分科目差距 > 1.0 时，在下一次推送中嵌入建议。

#### 3A-2 动态课表调整

| 指令 | 行为 |
|------|------|
| `#调整 [科目名] [时长]` | 当日增加该科目分钟数，自动缩减其余科目 |
| `#还原课表` | 清除当日所有 override |

#### 3A-3 遗忘曲线自动安排

mastery_level 上升后，按艾宾浩斯间隔更新 `next_review_at`：

```
1天 → 3天 → 7天 → 15天 → 30天
```

### B. 英语单词"空隙"推送

#### 推词触发条件（ALL 同时满足）

| # | 条件 |
|---|------|
| 1 | `is_idle = TRUE`（当前为空闲时段） |
| 2 | 沉浸模式未激活 |
| 3 | 情绪熔断器未触发 |
| 4 | 距上次推词 ≥ 8 分钟 |

#### 交互闭环

- ✅ 全部正确 → 降低词权重，`mastery_level +0.2`
- ⚠️ 部分正确 → 错误词入错题本，权重 ×1.5
- ❌ 超时/全错 → 存入错题本，22:30 晚间总结强制复现

错题本权重衰减：`weight = weight / (1 + 0.3 × correct_streak)`

### C. 知识点跨科串联

查询 `cross_links` 字段，复习某知识点时自动追加关联提醒。维护方式：

- `#关联 [知识点A] [知识点B]` → 双向写入
- CSV 导入时解析 `cross` 列自动写入
- Manager 每周日分析密度，提示用户补充关联

### D. 晚间复盘推送（22:30）

- 当日打卡覆盖率（各科分列）
- 单词成功率 + 错题本词汇列表
- 明日建议（按 `importance × (5 - mastery)` 降序排列 Top 3）
- 角色口吻的鼓励语

## 4. 情绪陪伴模块

> v3.0 核心新增。陪伴模块与学习调度模块**完全解耦**——陪伴会话期间，Scheduler 的推送自动进入挂起队列，待会话结束后补发或丢弃。

### 4.1 触发机制

情绪陪伴通过三条路径触发，触发后统一进入**陪伴会话状态**（`companion_mode = TRUE`）。

#### 路径一：用户主动倾诉（user_confide）

Emotion Detector 对用户消息进行轻量关键词匹配（不调用外部 API，本地规则库）：

```
情绪信号词库（可扩展）：
  焦虑类：好烦 / 烦死了 / 压力好大 / 好慌 / 睡不着 / 好难 / 坚持不住了
  沮丧类：好差 / 完了 / 好绝望 / 没意思 / 不想学了 / 放弃
  疲惫类：好累 / 累了 / 撑不住 / 身体不行了
  求陪伴类：有点难受 / 想聊聊 / 陪我说说话
```

匹配到任意信号词时，系统不直接回复学习内容，而是：

1. 暂停当前推送队列
2. 切换至陪伴会话模式
3. 用当前激活角色的**情绪接纳话术**响应

> **设计原则**：Emotion Detector 宁可漏检，不可误触发。情绪陪伴打断学习节奏，只有中高置信度（≥2 个信号词，或 1 个强信号词）才自动触发。

#### 路径二：系统检测异常（system_detect）

以下异常状态由 Scheduler 上报至 Emotion Detector：

| 异常类型 | 触发条件 | 陪伴优先级 |
|----------|----------|-----------|
| 情绪熔断 | 连续 3 次推词正确率 < 50% | 高，立即触发 |
| 长时间离线 | 计划学习时段内 > 90 分钟无任何消息 | 中，轻柔询问 |
| 深夜异常活跃 | 00:00–04:00 期间持续发消息 | 中，关怀提醒 |
| 连续多日缺卡 | 连续 3 天打卡覆盖率 < 30% | 高，专项关怀 |

#### 路径三：用户主动指令（command）

```
#陪我聊          → 直接进入陪伴会话，角色发起开场
#陪我聊 [内容]   → 带着话题进入会话
```

### 4.2 陪伴会话状态机

```
[正常模式]
    │
    ├─ 路径一/二/三 触发
    ▼
[陪伴会话中] ←─────────────────────────────┐
    │                                        │
    ├─ 角色发起情绪接纳响应                  │
    ├─ 用户持续对话（Scheduler 推送挂起）     │
    ├─ 角色根据对话内容动态调整响应策略       │
    │   · 倾听 → 共情 → 轻度引导 → 收尾     │
    │                                        │
    ├─ 用户发送 #结束聊天                    │
    ├─ 用户发送学习相关指令（视为主动退出）  │
    ├─ 会话静默超过 15 分钟                 │
    ▼
[冷却期（30分钟）] ──────────────────────────┘
    │  冷却期内不再主动触发情绪陪伴
    ▼
[正常模式]
```

### 4.3 陪伴对话策略（四阶段）

无论哪种角色，情绪陪伴对话均遵循以下四阶段结构，**只是语言风格不同**：

| 阶段 | 目标 | 持续轮数 |
|------|------|---------|
| **接纳** | 不评判地确认对方情绪，不急于给建议 | 1–2 轮 |
| **倾听** | 用开放式问题引导对方说出更多 | 2–4 轮 |
| **轻度引导** | 在对方有意愿时，提供视角或小建议，不强推 | 1–2 轮 |
| **收尾** | 给出当下可执行的一个小动作，温柔结束 | 1 轮 |

> 不主动追问超过 2 次，不强行给"解决方案"，不说教。

## 5. 人物卡系统

> 人物卡绑定全局推送语气。选定角色后，早安提醒、推词消息、进度报告、情绪陪伴的所有输出均使用对应口吻。

### 5.1 角色选择指令

```
#选择角色          → 展示角色列表（含简介）
#选择角色 [角色id]  → 切换角色，立即生效
#当前角色          → 查看当前激活角色信息
```

角色切换后发送确认消息，使用新角色的口吻说一句"自我介绍"。

### 5.2 人物卡规格说明

每张人物卡包含以下参数：

| 参数 | 说明 |
|------|------|
| `id` | 角色唯一标识（英文） |
| `name` | 角色名称 |
| `archetype` | 风格类型 |
| `tagline` | 一句话简介（展示在角色列表） |
| `tone_profile` | 口吻参数（见下） |
| `daily_scripts` | 日常推送脚本变体 |
| `emotion_scripts` | 情绪陪伴响应脚本 |

`tone_profile` 参数：

```yaml
tone_profile:
  address: ""          # 对用户的称呼方式
  self_ref: ""         # 自称
  punctuation: []      # 惯用标点/语气词
  catchphrase: []      # 口癖短句（随机插入）
  emoji_set: []        # 惯用 emoji
  verbosity: low/mid/high  # 话量
  formality: casual/neutral/formal  # 正式程度
```

## 11. 关键设计原则

**① 低干扰优先**  
沉浸模式激活时，所有非单词推送自动延后；情绪陪伴结束后，被挂起的推送选择性补发或丢弃，不堆积轰炸。

**② 情绪感知 > 任务驱动**  
Emotion Detector 的触发阈值宁可保守。陪伴是手段，不是功能，不应该成为新的打扰来源。

**③ 角色一致性**  
Persona Engine 渗透所有推送出口，同一角色在任何场景下的口吻应当一致，不出现"学习模式说话像零绮，情绪陪伴突然变白泉"的割裂感。

**④ 全学科平权**  
掌握度差距驱动调度，不偏袒任何科目，不硬编码课表优先级。

**⑤ 可扩展架构**  
新增科目只需 subjects 表插一条记录 + 上传 CSV；新增角色只需在角色库中追加一张人物卡 JSON，无需修改调度逻辑。

**⑥ 数据本地化**  
全部数据存于本地 SQLite，emotion_log 亦仅用于本地调度优化，不上传任何情绪数据。
