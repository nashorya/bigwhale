## 0. 系统初始化：目标院校与专业课确认

> 本节为系统首次启动时的必经流程，在任何知识点库加载、课表配置、日程生成之前完成。
> 初始化结果写入 `target_schools` 表，全程驱动后续专业课分配与打卡逻辑。

---

### 0.1 初始化触发时机

以下情况触发初始化引导流程：

- 用户首次启动系统（`user_init_complete = FALSE`）
- 用户主动发送 `#重新初始化`
- 用户发送 `#添加目标院校`（追加模式，不清空已有数据）

---

### 0.2 初始化对话流程

系统通过多轮对话引导用户完成填写，**不使用表单**，全程 QQ 消息交互。

#### 第一步：填写目标院校与专业

```
【上岸 初始化 · 第1步 / 4】

你好！在开始之前，我需要了解你的备考目标。

请告诉我你的第一志愿：
  院校名称：（如：北京大学）
  报考专业：（如：计算机科学与技术）

直接回复即可，格式：[院校名] [专业名]
例如：北京大学 计算机科学与技术
```

用户回复后，系统记录并询问是否有多个目标院校：

```
已记录：北京大学 · 计算机科学与技术 ✓

你是否还有其他目标院校？（备选志愿）
  · 有 → 回复院校名和专业名，格式同上
  · 没有 → 回复 [完成]

（最多支持3所目标院校，第一所为主目标）
```

#### 第二步：确认考试科目

系统根据专业名称**匹配已知规则**（公共课部分可自动填写），专业课部分要求用户手动填写：

```
【上岸 初始化 · 第2步 / 4】

根据你的报考信息，公共课科目如下，请确认：

  ✓ 政治（思想政治理论）
  ✓ 英语一
  ✓ 数学一
  （以上为计算机科学与技术通用科目，如有不同请告知）

现在请填写专业课科目：
  北京大学 · 计算机科学与技术 的专业课是？

  常见情况：
    · 408统考（数据结构+组原+操作系统+计算机网络）
    · 院校自命题（请填写科目名称，如：数据结构与算法）

  直接回复科目名称即可。
  不确定？发送 [#跳过] 先继续，之后可用 [#修改专业课] 补充。
```

用户填写后系统确认：

```
已记录专业课：408统考 ✓

北京大学 · 计算机科学与技术 初始科目配置：
  ① 思想政治理论
  ② 英语一
  ③ 数学一
  ④ 408统考

确认无误请回复 [确认]，需要修改请回复 [修改]。
```

#### 第三步：多目标院校专业课差异处理

若用户填写了多所院校，系统逐一比对各院校的专业课科目：

```
【上岸 初始化 · 第3步 / 4】

你填写了 2 所目标院校，我来对比一下专业课情况：

  主目标  北京大学   CS  →  ④ 408统考
  备选    上海交通大学 CS  →  ④ 数据结构（自命题）

⚠️ 两所院校专业课不同。

系统规则：
  · 日常打卡与学习计划 → 以主目标（北京大学）的专业课为准
  · 备选院校的不同专业课 → 不纳入日程调度，不推送打卡提醒
  · 你可以手动用 [#打卡] 指令为备选专业课打卡，系统会记录但不主动推送

如需为备选院校单独上传知识点，发送 [#同步 文件名 --school 上海交通大学]
```

#### 第四步：考试日期与备考阶段确认

```
【上岸 初始化 · 第4步 / 4】

最后一步，请填写你的考试日期：

  全国统考日期通常为每年12月第三周周六
  （2025年为 2025-12-20，2026年为 2026-12-19）

  直接回复日期，格式：YYYY-MM-DD
  例如：2025-12-20

  不确定具体日期？回复 [统考] 系统自动设置为当年全国统考日期。
```

确认后系统推送初始化完成总结：

```
🎉 初始化完成！

你的备考配置：

  主目标  北京大学 · 计算机科学与技术
    科目：政治 · 英语一 · 数学一 · 408统考

  备选    上海交通大学 · 计算机科学与技术
    科目：政治 · 英语一 · 数学一 · 数据结构（自命题，不纳入主调度）

  考试日期：2025-12-20（距今 XXX 天）
  当前备考阶段：基础期

接下来：
  · 发送 [#同步 文件名.csv] 上传你的专业课知识点
  · 发送 [#选择角色] 选择你的陪伴角色
  · 发送 [#今日计划] 查看今天的学习安排

准备好了吗？
```

### 0.3 专业课变更处理规则

用户通过 `#修改专业课 [院校名]` 触发专业课变更流程。

#### 变更判断逻辑

```
旧专业课科目集合  old_subjects
新专业课科目集合  new_subjects

相同科目  = old_subjects ∩ new_subjects  →  保留全部知识点与打卡记录，无需操作
不同科目  = old_subjects - new_subjects  →  系统提示用户手动决定是否删除
新增科目  = new_subjects - old_subjects  →  引导用户上传新科目知识点 CSV
```

#### 变更通知示例

```
【专业课变更确认】

检测到主目标专业课变更：
  旧配置：408统考
  新配置：数据结构（自命题）+ 操作系统（自命题）

科目对比结果：

  ✅ 保留（科目存在重叠或包含关系）：
     · 数据结构相关知识点（共 43 条）→ 保留

  ⚠️ 需要你手动处理（旧科目已不在新配置中）：
     · 计算机组成原理（共 31 条知识点）
     · 计算机网络（共 28 条知识点）

  以上知识点暂时保留在数据库中，但不再纳入日程调度。
  确认删除请发送：[#删除科目 计算机组成原理]
  确认保留请发送：[#停用科目 计算机组成原理]（保留数据但不推送）

  📥 新科目需要上传知识点：
     · 操作系统（自命题）→ 发送 [#同步 文件名.csv --subject 操作系统]
```

#### 三种处理状态说明

| 状态 | 指令 | 效果 |
|------|------|------|
| 保留并激活 | 自动（重叠科目） | 知识点保留，继续纳入日程调度 |
| 停用不删除 | `#停用科目 [科目名]` | 知识点保留在数据库，不推送、不调度；可随时重新激活 |
| 彻底删除 | `#删除科目 [科目名]` | 知识点从数据库删除，打卡记录保留在 `checkin_history` 存档表 |

> **设计原则**：系统永远不主动删除用户数据。变更时只提示、不自动清理，所有删除操作需用户明确确认。

### 0.4 多目标院校打卡规则汇总

| 场景 | 打卡行为 | 日程推送 |
|------|----------|---------|

| 公共课（政治/英语/数学） | 所有院校共享，正常打卡 | ✅ 主动推送 |

## 1. 专业分类策略

### 1.1 两类专业的本质差异

根据考研专业课是否有**全国统一考纲**，将十大热门专业分为两类：

```
A 类（统考/联考，可预置知识点库）
│
├─ 有教育部统一大纲，考点明确
├─ 系统内置完整考纲框架
└─ 用户可在框架节点下补充自己的笔记/细节

B 类（各校自命题，无法预置）
│
├─ 无全国统一考纲，各校出题范围差异极大
├─ 系统仅提供通用参考框架（常考方向）
└─ 用户必须结合目标院校大纲自行上传知识点 CSV
```

### 1.2 分类结果

| 类型 | 专业/科目 | 说明 |
|------|-----------|------|
| **A类** | 政治（思想政治理论） | 统考，大纲每年更新 |
| **A类** | 英语一 / 英语二 | 统考，词汇+题型固定 |
| **A类** | 数学一 / 数学二 / 数学三 | 统考，考纲稳定 |
| **A类** | 408（计算机联考） | 统考，四门合卷 |
| **A类** | 教育学（311统考） | 统考，五门合卷 |
| **A类** | 临床医学（306统考） | 统考，多门合卷 |
| **A类** | 管理类联考（199） | 联考，数学基础+逻辑+写作 |
| **B类** | 电子信息工程专业课 | 各校自命题，差异大 |
| **B类** | 材料科学与工程专业课 | 各校自命题，差异大 |
| **B类** | 机械工程专业课 | 各校自命题，差异大 |
| **B类** | 法学（学硕专业课） | 各校自命题；法律硕士联考另行处理 |
| **B类** | 经济学专业课 | 部分院校联考，多数自命题 |

> **法学特殊说明**：法学有两条路径——**法学学硕**（考两门院校自命题专业课）和**法律硕士联考**（考法综+法律硕士综合，有全国大纲）。系统初始化时需让用户选择方向，法律硕士联考路径可升级为 A 类处理。

## 3. 预置知识点库框架（A类专业）

### 3.1 库结构说明

每个 A 类科目的知识点库由两层组成：

```
syllabus_nodes（考纲节点，系统预置，只读）
└─ 一级章节：如「数据结构」
   └─ 二级考点：如「二叉树的遍历」
         ↓ 关联
knowledge_points（用户知识点，可读写）
└─ 用户在考点下创建的具体复习条目
   如「二叉树前序/中序/后序遍历的递归与非递归实现」
```

系统只预置到**二级考点**，用户在二级考点下自由添加具体复习条目、错题、笔记等。

## 5. 每日学习计划自动分布算法

### 5.1 设计目标

将所有科目的知识点，按照**考试距离 + 掌握度差距 + 遗忘曲线**三个维度，自动分配到每天的学习时段，生成当日知识点清单。

### 5.2 核心参数

| 参数 | 来源 | 说明 |
|------|------|------|
| `exam_date` | 用户初始化时填写 | 考试日期，全局参数，驱动所有时间计算 |
| `days_left` | `exam_date - today` | 剩余天数，实时计算 |
| `daily_capacity` | 用户设置 | 每日可学习总分钟数（默认 360 分钟） |
| `subject_ratio` | Manager 动态计算 | 各科目时间分配比例 |
| `priority_score` | 每个知识点独立计算 | 决定今天复习哪些知识点 |

### 5.3 知识点优先级评分

每天凌晨 00:05，系统为所有知识点重新计算当日优先级分数：

```python
def calc_priority(kp, days_left):
    # 基础分：重要程度越高、掌握度越低，分越高
    base = kp.importance * (6 - kp.mastery_level)

    # 遗忘系数：距上次复习越久，分越高（最大2倍）
    days_since = (today - kp.last_review_at).days if kp.last_review_at else 30
    forget_factor = min(2.0, 1 + days_since / 14)

    # 逾期系数：next_review_at 已过期的知识点额外加权
    overdue_factor = 1.5 if kp.next_review_at < today else 1.0

    # 冲刺系数：考试 ≤30天时，高频考点（importance=3）额外加权
    sprint_factor = 1.3 if days_left <= 30 and kp.importance == 3 else 1.0

    return base * forget_factor * overdue_factor * sprint_factor
```

### 5.4 每日计划生成流程

```
① 按科目时间比例，计算各科今日可分配知识点数量
   （每个知识点预估复习时间：importance=1 → 10min，2 → 15min，3 → 20min）

② 在每科内，按 priority_score 降序排列，取前 N 个知识点

③ 若某科今日知识点全部 mastery_level ≥ 4，自动补入其他落后科目

④ 生成当日清单，写入 daily_plan 表（见第7节）

⑤ 早安推送时将清单发给用户，格式见下方示例
```

### 5.5 备考阶段自动切换

系统根据 `days_left` 自动调整调度策略：

| 阶段 | 剩余天数 | 调度策略 |
|------|----------|---------|
| 基础期 | > 120天 | 均衡推进，新知识点为主，间隔复习为辅 |
| 强化期 | 60–120天 | 加大 importance=2/3 知识点比例，开始系统性错题回顾 |
| 冲刺期 | ≤ 60天 | importance=3 知识点优先级提升30%，停止引入新知识点 |
| 最后冲刺 | ≤ 14天 | 只复习 mastery_level ≤ 3 的知识点，不引入任何新内容 |

## 6. 打卡正反馈系统

> 反馈分两层：**单次打卡后立即推送（简短）** + **当日所有打卡完成后统一推送（完整）**

```

**纪律**
```
[B树与B+树] 记录。4分。

   · 计划 / 完成知识点数（各科分列）

   · 掌握度提升的知识点列表（"今天这些都变强了"）

## 9. 人物卡文件系统

### 9.1 存放位置与加载机制

人物卡采用**JSON 文件**管理，与代码仓库放在一起，Persona Engine 启动时全量加载进内存。数据库的 `persona_config` 表只存"当前激活哪张卡"的状态索引，不存卡的内容本身。

```
/上岸
  /personas
    /builtin                ← 内置人物卡（系统维护，只读）
      lingqi.json           · 零绮·冷静理性系

      suwan.json            · 苏晚·温柔治愈系
      jilv.json             · 纪律·毒舌激励系
    /custom                 ← 用户自定义卡（v3.3 启用，暂为空目录）
      .gitkeep
    personas_index.json     ← 角色索引文件（id、名称、简介、文件路径）
```

### 9.4 Persona Engine 渲染流程

所有对外推送消息经过以下流程输出：

```
① Scheduler / Manager 生成原始模板消息
   如："[{kp_name}] 已记录。掌握度 {before}→{after}。"

② Persona Engine 读取 persona_config.active_persona
   确认当前激活角色 id（如 lingqi）

③ 从已加载的角色 JSON 中匹配对应 script key
   如：checkin_scripts.single_immediate

④ 填充模板变量（kp_name、before、after 等）

⑤ 注入 tone_profile 修饰
   · 随机插入 catchphrase（概率 20%，避免每次都出现）
   · 按 emoji_set 决定是否附加 emoji
   · 按 verbosity 决定是否截断或扩展内容

⑥ 输出最终消息 → 通信层发送
```

### 9.5 新增/修改人物卡

**修改内置卡**：直接编辑 `/personas/builtin/` 下对应 JSON 文件，重启服务后生效。无需修改任何业务代码。

**新增内置卡**：
1. 在 `/personas/builtin/` 下新建 JSON 文件（参照 9.2 结构）
2. 在 `personas_index.json` 的 `builtin` 数组中追加索引条目
3. 重启服务

**v3.3 自定义卡（预留）**：用户上传 JSON 文件至 `/personas/custom/`，系统读取后写入 `custom_personas` 数据库表，通过 `#上传角色卡` 指令触发。内置卡与自定义卡的渲染流程完全相同，Persona Engine 统一处理。
